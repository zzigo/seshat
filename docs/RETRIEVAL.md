# Retrieval architecture

Seshat uses one stable UUID per Docling chunk as the bridge across every retrieval layer.

1. PostgreSQL is the source of truth and provides exact lexical search through a generated
   `tsvector` and GIN index.
2. Qdrant stores dense vectors generated by Ollama. It is optional: extraction and lexical
   search continue when Qdrant is unavailable, and pending chunks are indexed when it returns.
3. PostgreSQL stores evidence graph nodes and edges. Neo4j is an optional mirror configured
   through HTTP; Seshat does not depend on Neo4j for correctness.
4. The web retrieval layer fuses lexical, vector and graph rankings with weighted RRF and then
   rechecks every chunk against PostgreSQL access rules.
5. The optional reasoning action sends only revalidated evidence to Ollama and requires inline
   numbered citations. Retrieval remains usable when the model is unavailable.

Kùzu is deliberately not supported because its upstream repository was archived in October 2025.

## Environment

```dotenv
QDRANT_URL=http://127.0.0.1:6333
QDRANT_API_KEY=replace-with-a-long-random-secret
QDRANT_COLLECTION=seshat_chunks
OLLAMA_EMBED_MODEL=nomic-embed-text
VECTOR_BATCH_SIZE=8

# Optional graph mirror
NEO4J_URL=http://127.0.0.1:7474
NEO4J_USER=neo4j
NEO4J_PASSWORD=replace-me
```

Without `QDRANT_URL`, vector indexing is deferred and `/search` remains fully usable in lexical
and graph modes. Without `NEO4J_URL`, the evidence graph remains in PostgreSQL.

## Qdrant

```sh
QDRANT_API_KEY="$(openssl rand -hex 32)" \
SESHAT_RETRIEVAL_DATA=/mnt/seshat-hot/retrieval \
docker compose -f ops/compose.retrieval.yml up -d
```

The Qdrant collection is created lazily using the dimensionality returned by the configured
Ollama embedding model. Vector payloads contain only retrieval metadata; chunk text and access
control remain authoritative in PostgreSQL.

## Existing documents

The backfill reuses the `chunks.jsonl` artifacts already stored in Wasabi; it does not send PDF or
EPUB files through Docling again. Inspect first, then apply:

```sh
npm run build:packages
npm run backfill:retrieval
npm run backfill:retrieval -- --apply --queue-relate
```

`--queue-relate` also rebuilds the graph and should be used while the worker is active. Vectors
remain `pending` until a worker with Qdrant configured indexes them in gradual batches.

## Reindexing

Re-extracting a document replaces its PostgreSQL chunks. Stable IDs are deterministic for a
reference, ordinal and exact chunk content. New or changed chunks enter the vector queue with
`pending` status. Failed vector batches retry at most three times until explicitly reset.
Replaced chunks enter a durable PostgreSQL deletion queue, so stale Qdrant points are removed
after an outage instead of becoming orphaned vectors.
